1. 这是什么

unity-mcp-beta 是 CoplayDev 维护的 Unity 编辑器 MCP（Model Context Protocol）插件，让 Claude 等 LLM 通过统一协议直接驱动 Unity 编辑器：读场景、改 GameObject、管资源、跑测试、出包、执行 C# 等。架构由两部分组成：

  • Python FastMCP 服务器（Server/src/）：暴露 MCP 工具给客户端（Claude Desktop / Cursor / Windsurf）。
  • Unity 编辑器侧 C# 包（MCPForUnity/）：通过 WebSocket（或 stdio）与 Python 端通信，执行真正的编辑器操作。

cocosMcp 的核心架构就是镜像这一套：Python 端几乎可以原样复用，Cocos 端用 JS 替代 C#。

2. “生成一份 HTML 文档” 在 unity-mcp 里是什么

结论：unity-mcp 并不生成本地 HTML 文档。Server/src/cli/commands/docs.py 提供的是 unity-mcp docs get <Class> [Member] 这条 CLI 子命令，它做的事情是从 Unity 官方 ScriptReference 在线抓取那个类/成员的文档页（URL 拼成 https://docs.unity3d.com/ScriptReference/Class.Member.html），输出可以是 text/json/table 三种格式。

实际负责抓取的工具是 services/tools/unity_docs.py，它对外是一个 MCP tool，提供 4 个 action：

action含义
get_doc取某个 C# API 类/成员的 ScriptReference 页面。
get_manual取 Unity Manual 的某一页。
get_package_doc取某个包（如 URP、Cinemachine）的文档。
lookup合并搜索（doc + manual + package）。

对应到 cocos 2.4 的等价做法（如果想做）：抓 docs.cocos.com/creator/2.4/manual/en/ 和 docs.cocos.com/creator/2.4/api/en/ 这两棵树。注意 Cocos 引擎 API 本身是 JS，所以更接近“查在线手册”而不是“反射 C# 类型”。

如果 liyu 你的原意是“让 cocosMcp 也输出一份项目 HTML 报告（列出场景、节点、资源……）”，这在 unity-mcp 里其实是没有的——unity-mcp 只暴露 MCP 工具，让 LLM 自己组织报告。要做这个能力 cocosMcp 得自己加一个 generate_project_report 之类的工具。

3. unity-mcp 的 MCP 工具全集（Python 端）

每个工具是一个 @mcp_for_unity_tool 装饰的 async 函数，行为通过 action 参数分支。下面给出工具名 + 官方描述（直引），按主题分组。

3.1 场景 / GameObject 类

manage_scene场景 CRUD：get_hierarchy, get_active, get_build_settings, get_loaded_scenes, scene_view_frame, create, load, save, close_scene, set_active_scene, move_to_scene, validate。
manage_gameobjectGameObject CRUD：create / modify / delete / duplicate / move_relative / look_at。不负责搜索（用 find_gameobjects）和组件（用 manage_components）。

5. 通讯架构

主链路（HTTP/WS）

  • Python 起一个 Starlette 服务，把 /ws/mcp-bridge 暴露为 WebSocket 端点（PluginHub）。
  • Unity 插件主动连过来，发 RegisterMessage（session_id、项目元数据）。
  • Python 把连接存进 _connections[session_id]。
  • MCP 工具调用 → 发 ExecuteCommandMessage{command_type, params, command_id}。
  • Unity 执行对应 C# handler，回 CommandResultMessage，按 command_id 配回 Future。
  • 10s 一次 ping/pong；默认 30s 超时，read_console/ping 等只给 2s 快失败。
  • 支持 retry_on_reload：domain reload 重连后自动重试。

旧链路（stdio/TCP）

  • transport/legacy/unity_connection.py + port_discovery.py。
  • 本地起 TCP server，发现 Unity 监听端口后建池。
  • unity_transport.send_with_unity_instance() 按 config.transport_mode 路由到 HTTP 或 stdio。

消息类型

Welcome / Register / RegisterTools / ExecuteCommand / CommandResult / Ping / Pong / SessionList / SessionDetails。全部 JSON。

对比 cocosMcp：当前只实现了一个简化版的“WS 一帧一 JSON”，没有 session 概念、没有多实例、没有 ping/pong（依赖 ws 库默认），没有 retry_on_reload。这些都是后续可选项。

6. 设计哲学（值得复制到 cocosMcp 的）

  • 对称命名：Python tool 名 = bridge command = C# handler 文件名。cocosMcp 已经在 CLAUDE.md 里立了这条规矩。
  • 单工具单职责，靠 action 分支：避免参数膨胀；超过 ~6 个 action 拆。
  • 读用 resource，写用 tool：unity-mcp 把“查询场景树”做成 resource URI，cocosMcp 现在全是 tool。这是 cocosMcp 当前的一个差距。
  • 错误如实回流：bridge 失败时 Python 端返回的不是抛异常，而是 {success: false, error: ...}。
  • 反射防幻觉：unity_reflect 让 LLM 在写代码前先确认 API 真的存在。cocosMcp 可以做同样的 cc_reflect。
  • preflight 拦破坏性：编译中/测试中不让破坏性工具跑。
  • execute_code 是 last resort：但要带 history/replay，方便 LLM 复盘和用户审计。
  • 工具分组按 session 启停：把权限控制做到协议层。
